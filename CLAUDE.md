# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Volante is a desktop focus timer built as a Tauri 2 app (Rust shell + React 19/TypeScript frontend). It frames deep-work sessions as F1 races: a car drives around a real circuit's SVG path for the duration of a session, the user can activate "regulations" (Boost/Overtake/DRS) to temporarily speed up the timer, and breaking focus (pausing, unfocusing the window, going idle) incurs penalties that subtract progress. All data is local-only — no network, no telemetry.

## Commands

```bash
npm run tauri dev      # Run the full desktop app (Rust + Vite frontend, port 1420)
npm run dev            # Vite frontend only (browser; Tauri APIs fall back to browser equivalents)
npm run build          # Type-check (tsc) + production Vite build
npm run tauri build    # Full production bundle → src-tauri/target/release/bundle/
npm run test           # Run vitest once
npm run test:watch     # Vitest watch mode
npx vitest run src/engine/regulationsEngine.test.ts   # Run a single test file
```

Tests run in jsdom (see `vitest.config.ts`). `npm run build` will fail the build on unused locals/params — TS is in strict mode with `noUnusedLocals`/`noUnusedParameters`.

## Architecture

The codebase enforces a strict separation between **pure logic**, **state**, **React glue**, and **persistence**. When changing behavior, change it at the right layer — do not, for example, put filesystem calls in stores or season-data imports in `sessionStore`.

### Layers (data flows top → bottom)

- **`src/engine/`** — Pure, side-effect-free logic. No store access, no React, no I/O. Each module has a colocated `.test.ts`. This is where game rules live:
  - `sessionStateMachine.ts` — the canonical state transition table (`setup → running → paused/completed/abandoned`). `sessionStore` duplicates this guard internally; keep them in sync.
  - `regulationsEngine.ts` — can-activate checks, button visual state, cooldown/usage math, interruption-penalty calculation.
  - `penaltyDetector.ts` — `createIdleDetector` (throttled DOM activity listeners) and `createUnfocusDetector` (Tauri `onFocusChanged`, falls back to browser blur/focus when not in Tauri). Returns start/stop/destroy handles; does NOT touch the store.
  - `timer.ts` — `requestAnimationFrame` loop producing `deltaMs` callbacks.
  - `progressCalculator.ts` — progress/lap math.

- **`src/stores/`** — Zustand stores hold all global runtime state.
  - `sessionStore.ts` is the runtime "brain": owns the active `Session`, all lifecycle transitions, `tick(deltaMs)` (advances `effectiveProgressSec` by `deltaSec * currentPaceMultiplier`, expires regulations, auto-completes at 100%), regulation activation, and penalty application. It deliberately does NOT import season data or call the filesystem.
  - **Regulation activation gotcha**: `activateRegulation()` on the store is a low-level stub with hardcoded defaults. Real activation goes through the exported `activateRegulationWithConfig(type, paceMultiplier, durationSec, cooldownSec)` function, which the `useRegulations` hook calls after validating with the engine. Cooldown starts when the regulation *ends*, not when it starts.
  - `historyStore.ts` — past sessions, persisted to `history.json`, capped at 100 entries (oldest pruned).
  - `settingsStore.ts` — user settings, persisted to `settings.json`.

- **`src/hooks/`** — React glue that wires engine + stores together. `useTimer` drives the store's `tick` from the timer engine reacting to session state; `useRegulations` exposes button state/actions; `usePenaltyDetection` wires the detectors to `applyPenalty`. Each has colocated tests.

- **`src/utils/storage.ts`** — the ONLY module that touches the filesystem. Wraps `@tauri-apps/plugin-fs`, reading/writing JSON files (`settings.json`, `history.json`) in the Tauri AppData dir. Filenames are validated against path traversal. Only stores call it (via the history/settings stores), never engine or components.

- **`src/screens/`** + **`src/components/`** — UI. Screens are full-page views (Setup → Race → Summary); `App.tsx` picks the view purely from `session.state` (there is no router). Each component lives in its own folder with a colocated `.module.css`.

### Data

- **`src/data/seasons/`** — `SeasonRuleset` definitions (`season2025` = DRS + Overtake, `season2026` = Boost + Overtake). **`getSeasonByYear()` is not a plain lookup** — it clones the base ruleset and rewrites regulation durations/cooldowns/limits from user settings (relative-percent vs absolute), then clamps Boost/DRS duration ≤ Overtake duration. Always go through it rather than reading `season20xx` directly.
- **`src/data/tracks/`** — 24 circuits. `trackPaths.ts` holds extracted SVG `d` path strings; `trackCatalog.ts` holds metadata (lap time, start offset, accent color). Paths were generated by `scripts/extract-track-paths.ts` from the f1-circuits-svg dataset.

### Car-on-track rendering

`src/utils/interpolatePath.ts` uses the browser SVG DOM API `getPointAtLength()` to compute the car's (x, y, angle) along the track path. This requires a *rendered* `<path>` element, so `TrackRenderer` passes a ref to the live SVG node — this logic cannot run server-side or in pure tests against the path string alone.

## Conventions

- TypeScript strict mode, no `any`. CSS Modules only (one `.module.css` per component); shared design tokens are CSS variables in `src/index.css`. PascalCase components (one per file, functional + hooks), camelCase functions/vars.
- Tests are colocated next to source (`foo.ts` + `foo.test.ts`), using vitest + Testing Library.
- Many source files reference an `ARCHITECTURE.md` (e.g. "From ARCHITECTURE.md §9.5") in their headers. That file is not currently in the repo — the section numbers are historical breadcrumbs, not a present source of truth.
- Tauri identifier is `com.volante.app`; capabilities (`src-tauri/capabilities/default.json`) grant only `core:default` and `fs:default`. Adding a Tauri plugin/API generally means adding a permission here.
- PRs target `main`; commit messages follow conventional prefixes (`feat:`, `fix:`, `perf:`).
